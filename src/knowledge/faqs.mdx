### What is your preferred tech stack and why?

My happy path is **TypeScript + Node.js + Next.js + MongoDB** for most web applications. I reach for this stack because:
- TypeScript catches entire classes of bugs at authoring time, not in production at 2am
- Node.js event loop model is genuinely efficient for I/O-heavy workloads
- Next.js gives you sensible defaults for routing, SSR, and deployment without fighting the framework
- MongoDB's flexible schema and powerful aggregation pipeline cover 80% of data access patterns I encounter

I'll always pick the right tool for the actual problem though — I've used Supabase (Postgres) for Mintmark because row-level security was genuinely the right primitive there.

### What kind of projects excite you most?

Backend systems with real performance stakes — something where you have to think about latency, concurrency, or financial correctness. Real-time systems. Payment infrastructure. Anything where the constraints are interesting and the bugs are expensive.

On the product side, developer tools and productivity software. I built Mintmark partly because I would use it myself every day.

### What's your working style — sync or async?

### How do you handle code reviews?

I give specific, actionable feedback tied to a reason. "This could be a performance issue because [X]" beats "this seems slow." I don't comment on style unless the linter isn't configured to catch it. I try to distinguish between blocking concerns and optional suggestions.

I expect the same in return.

### What is MintMark?

Mintmark is a personal branding platform I'm building for developers. The core idea: you learn things every day — you read articles, solve problems, build features, explore new libraries — but very little of that learning becomes visible to anyone. Mintmark connects to where your learning already happens (GitHub activity, LeetCode, your own notes) and turns it into platform-ready content for LinkedIn, X, and Medium, simultaneously.

It also has a personal knowledge assistant — a RAG-powered chatbot that answers questions strictly from your own notes and posts, not the general internet.

---

### Why did you build MintMark?

I kept telling myself I'd post more on LinkedIn "when I had something worth saying" and then not posting because the activation energy was too high. I figured if I was feeling that friction, other developers probably were too. So I built the tool I needed.

It also became the project where I went deepest on RAG systems, background job orchestration, BYOK AI integration, and multi-tenant architecture — all things I wanted production experience with.

---

### What problem does MintMark solve exactly?

The gap between "I learned something today" and "I published something today." Most developers have consistent learning activity (commits, problems solved, articles read) but almost none of it becomes public content. Mintmark closes that gap by automating the content generation step while keeping you in control of what actually gets published.

### What does your debugging process look like?

Reproduce it reliably first. Without a reproduction, you're guessing. Then isolate the smallest possible failing case. Read the actual error — the full stack trace, not just the last line. Form a hypothesis, test it, move on. I don't reach for `console.log` until I've read the error message twice.

---

### Do you write tests?

Yes, and I regret every codebase where I didn't start with them. Integration tests over unit tests where possible — mocking too deep gives you false confidence. I set up CI to run tests on every PR and treat a broken test as a broken build, not a suggestion.

### What's your opinion on [hot take: tabs vs spaces]?

Tabs. But I'll defer to whatever the project's Prettier config says because this is not worth a discussion. The only correct answer is: pick one and enforce it with a formatter so no human has to think about it again.

### What's your opinion on JavaScript frameworks in general?

I think framework churn is overstated as a problem. The fundamentals — how the DOM works, what the event loop does, how HTTP requests are structured — stay stable. Frameworks are just opinions about how to organize that. Pick one that has good primitives, learn it deeply, and don't rewrite things that are working.

### What's one thing you wish you'd learned earlier?

That "working code" and "correct code" are meaningfully different and the gap between them is usually where production incidents live. I spent too long being satisfied with "it works on my machine." The shift to thinking about edge cases, concurrent access, and failure modes as first-class design concerns changed how I write code permanently.
