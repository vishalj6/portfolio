### What is your preferred tech stack and why?

My default is **TypeScript + Node.js + Next.js + MongoDB** for most web applications:

- TypeScript catches entire classes of bugs at authoring time, not in production at 2am
- Node.js event loop model is efficient for I/O-heavy workloads
- Next.js gives sensible defaults for routing, SSR, and deployment without fighting the framework
- MongoDB's flexible schema and aggregation pipeline cover most data access patterns I encounter

That said, I'll always pick the right tool for the problem. I use Supabase (Postgres) for Mintmark because row-level security was genuinely the right primitive there.

### What kind of projects excite you most?

On the product side: developer tools and productivity software. I built Mintmark partly because I would use it myself every day.

### What's your working style — sync or async?

Async for execution, sync for alignment. Written context over a meeting for anything that can be communicated clearly in text. Architecture discussions, ambiguous requirements, and retrospectives are worth getting everyone in a room. Everything else: a Slack thread will do.

---

### How do you handle code reviews?

Specific, actionable feedback tied to a reason. "This could be a performance issue because [X]" beats "this seems slow." I don't comment on style if the linter handles it. I distinguish blocking concerns from optional suggestions clearly.

I expect the same in return.

### What is Mintmark?

Mintmark is a personal branding platform I'm building. The core idea: you learn things every day — you read articles, solve problems, build features — but almost none of that learning becomes visible to anyone.

Mintmark connects to where your learning already happens (GitHub, LeetCode, your own notes) and turns it into platform-ready content for LinkedIn, X, and Medium. It also has a personal knowledge assistant — a RAG-powered chatbot that answers questions strictly from your own notes and posts, not the general internet.

---

### Why did you build Mintmark?

I kept telling myself I'd post more on LinkedIn "when I had something worth saying" — and then not posting because the activation energy was too high. If I was feeling that friction, others probably were too. So I built the tool I needed.

It also became the project where I went deepest on RAG systems, background job orchestration, BYOK AI integration, and multi-tenant architecture — all things I wanted real production experience with.

---

### What problem does Mintmark solve?

The gap between "I learned something today" and "I published something today." Most developers have consistent learning activity but almost none of it becomes public content. Mintmark closes that gap by automating the content generation step while keeping you in control of what actually gets published.

### What does your debugging process look like?

Reproduce it reliably first — without a reproduction, you're just guessing. Isolate the smallest failing case. Read the actual error: the full stack trace, not just the last line. Form a hypothesis, test it, move on. I don't reach for `console.log` until I've read the error message twice.

---

### Do you write tests?

Yes — and I regret every codebase where I didn't start with them. Integration tests over unit tests where possible: mocking too deep gives you false confidence. CI runs tests on every PR and a broken test is a broken build, not a suggestion.

---

### VS Code or something else?

### What's your debugging rubber duck?

Claude Code. It asks good clarifying questions instead of just nodding. Walking away for 20 minutes also works better than staring at the same bug harder.

---

### What's your opinion on tabs vs spaces?

Tabs. But I'll defer to whatever Prettier says — this is not a discussion worth having. Pick one, enforce it with a formatter, and never think about it again.

---

### What's your opinion on JavaScript framework churn?

Overstated as a problem. The fundamentals — how the DOM works, what the event loop does, how HTTP is structured — stay stable. Frameworks are just opinions about organizing that. Pick one with good primitives, learn it deeply, don't rewrite things that are working.

---

### What's your Stack Overflow usage like?

Mostly read, rarely post. Docs before Stack Overflow, Stack Overflow before AI for anything security-related. For everything else: whatever gets me the answer fastest. Pasting an error into Claude Code has a better hit rate these days for Node/TypeScript issues.

---

### What's one thing you wish you'd learned earlier?

That "working code" and "correct code" are meaningfully different — and the gap between them is usually where production incidents live. I spent too long being satisfied with "it works on my machine." Thinking about edge cases, concurrent access, and failure modes as first-class design concerns changed how I write code permanently.

---

### What do you do when you're stuck on a problem?

Explain it out loud — to a teammate, or to a rubber duck if no one's available. Walking away for 20 minutes works better than staring harder. Claude Code is my current rubber duck of choice.
